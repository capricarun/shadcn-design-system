Component Reference

Button

Interactive

Triggers an action or navigates. Use variant to communicate intent and size to establish hierarchy.

Variants
Sizes & States
HTML + CSS 
<!-- Variant: Default -->
<button class="btn btn-md btn-primary">Save changes</button>

<!-- Variant: Destructive -->
<button class="btn btn-md btn-destructive">Delete</button>

<!-- Variant: Outline -->
<button class="btn btn-md btn-outline">Cancel</button>

<!-- Disabled -->
<button class="btn btn-md btn-primary" disabled>Saving…</button>

/* CSS tokens used */
.btn-primary {
  background-color: hsl(var(--primary));
  color:            hsl(var(--primary-foreground));
}
.btn-primary:hover {
  background-color: hsl(var(--primary) / 0.88);
}
Prop / ClassTypeDefaultDescription
btn-primaryvariant—Uses --primary background
btn-destructivevariant—Uses --destructive for danger actions
btn-outlinevariant—Bordered, uses --border
btn-secondaryvariant—Subdued with --secondary
btn-ghostvariant—Transparent, hover shows --accent
btn-sm / btn-md / btn-lgsizebtn-md36px / 40px / 44px height
disabledHTML attrfalseApplies 50% opacity, blocks interaction
✓ Do
Use Default for the primary CTA per screen — one per view.
Use Outline or Ghost for secondary/cancel actions alongside a primary.
Use Destructive only for irreversible actions (delete, remove, revoke).
Always add disabled during async operations to prevent double-submit.
✕ Don't
Don't place two Default buttons side-by-side — creates visual ambiguity.
Don't use Destructive for routine actions like "clear filters" or "reset form".
Don't hard-code colours — always use --primary, --destructive tokens.
Don't use Link variant for navigation — use an <a> element instead.

Badge

Display

Small label for status, category, or count. Never interactive on its own.

Variants
Default Secondary Destructive Outline Success Warning
HTML
<span class="badge badge-default">New</span>
<span class="badge badge-secondary">Draft</span>
<span class="badge badge-destructive">Error</span>
<span class="badge badge-success">Published</span>
<span class="badge badge-outline">Beta</span>
✓ Do
Use Destructive for error/critical states, Success for live/active states.
Keep badge text under 3 words — it's a label, not a sentence.
Use Outline when the badge sits on a coloured background to avoid clashing fills.
✕ Don't
Don't make badges clickable without adding button semantics and a hover state.
Don't use multiple filled-colour badges in the same row — it creates noise.
Don't use badge for numeric counts > 99; show "99+" instead.

Input

Form

Single-line text entry. Always pair with a Label and optional helper/error text.

States
HTML
<!-- Default -->
<input class="input" type="email" placeholder="Email" />

<!-- Error state -->
<input class="input input-error" aria-invalid="true" />

<!-- Disabled -->
<input class="input" disabled />

/* Error modifier */
.input-error { border-color: hsl(var(--destructive)); }
✓ Do
Always associate a <label> using for/id — never rely on placeholder alone.
Add aria-invalid="true" and aria-describedby pointing to error text on error state.
Use type="email", type="password", type="number" for semantic correctness.
✕ Don't
Don't use placeholder text as the only label — it disappears on input.
Don't show an error state before the user has interacted (blur or submit first).
Don't nest inputs inside labels without explicit for/id — screen readers behave inconsistently.

Checkbox

Form

Binary toggle for independent options. Use for multi-select lists.

States
HTML
<label class="checkbox-wrap">
  <input type="checkbox" class="checkbox" id="terms" />
  Accept terms and conditions
</label>
✓ Do
Wrap in a <label> so clicking the text also toggles the checkbox.
Use for independent boolean options (e.g. "Send me updates").
Use a checkbox group for multi-select; use radio for single-select-from-many.
✕ Don't
Don't use a checkbox in place of a Toggle/Switch for live settings that apply immediately.
Don't style away the focus ring — keyboard users rely on it.
Don't pre-check consent or agreement checkboxes — explicit opt-in required by GDPR.

Switch

Form

Immediate on/off toggle. Use when the change takes effect instantly without a save step.

States
HTML
<label class="switch-wrap">
  <input type="checkbox" class="switch" role="switch" />
  Dark mode
</label>
✓ Do
Add role="switch" and aria-checked for screen reader clarity.
Apply changes immediately — Switch implies instant effect, unlike Checkbox in a form.
Always label the switch with what it enables (e.g. "Email notifications").
✕ Don't
Don't require a save button after toggling — that's what a Checkbox is for.
Don't use Switch for filtering or selection; use Checkbox for that.
Don't label the switch "On/Off" — describe what will be enabled.

Select

Form

Choose one option from a dropdown list.

Default
HTML
<label for="framework">Framework</label>
<select class="select" id="framework">
  <option value="">Select…</option>
  <option>React</option>
  <option>Vue</option>
</select>
✓ Do
Use Select for 5+ options. For 2–4, consider radio buttons for visibility.
Include a blank/placeholder option (e.g. "Select a country…") when no default is valid.
✕ Don't
Don't use Select for boolean choices — use a Switch or Checkbox.
Don't mix disabled and enabled options without explaining why options are unavailable.

Label

Form

Accessible caption for form controls. Always linked via for/id.

We'll send a confirmation link.
HTML
<label class="label" for="email">
  Email <span aria-hidden="true">*</span>
</label>
<input class="input" id="email" type="email" required />
✓ Do
Mark required fields with aria-required="true" and a visible indicator.
Place the label above the input, not beside it — easier to scan on mobile.
✕ Don't
Don't use placeholder text as the label — it disappears and fails accessibility audits.
Don't make labels all-caps — they become harder to read at small sizes.

Form Field

Composite

Compound of Label + Input + Helper/Error text. The building block for every form.

Valid vs Error
Must be unique across the platform.
Enter a valid email address.
HTML
<!-- Error state -->
<div class="form-field">
  <label class="label" for="email">Email</label>
  <input class="input input-error" id="email"
    aria-invalid="true"
    aria-describedby="email-err" />
  <span class="error-text" id="email-err">
    Enter a valid email.
  </span>
</div>
✓ Do
Always use aria-describedby pointing to the error/helper text id.
Show error messages below the field (not in a toast) so they're spatially linked.
Validate on blur, not on keypress — don't interrupt users while typing.
✕ Don't
Don't show all form errors in a single banner at the top — users lose context of which field failed.
Don't remove the error message until the user has corrected the value and re-blurred.

Card

Display

Surface for grouping related content. Composed of Header, Content, and Footer slots.

Account Settings
Update your profile information and preferences.
HTML
<div class="card">
  <div class="card-header">
    <div class="card-title">Title</div>
    <div class="card-desc">Description</div>
  </div>
  <div class="card-content"> … </div>
  <div class="card-footer">
    <button class="btn btn-sm btn-primary">Save</button>
  </div>
</div>
✓ Do
Use Card to visually group related information that belongs together logically.
Place primary actions in card-footer to establish a consistent reading flow.
✕ Don't
Don't nest cards inside cards — it creates ambiguous visual hierarchy.
Don't put unrelated content in the same card to "save space" — each card = one concern.

Avatar

Display

Represents a user with an image or fallback initials.

AK
JD
RK
MJ
HTML
<!-- Initials fallback -->
<div class="avatar avatar-md" role="img" aria-label="Arun Kumar">
  AK
</div>

<!-- With image -->
<div class="avatar avatar-md">
  <img src="photo.jpg" alt="Arun Kumar" />
</div>
✓ Do
Always have a fallback — show initials when the image fails to load.
Add aria-label with the user's name for accessibility.
✕ Don't
Don't use generic icons as fallbacks — initials are more identifiable.
Don't use avatars larger than avatar-xl in lists — they dominate layout.

Alert

Feedback

Inline callout for important contextual messages. Not dismissible by default.

ℹ️
Heads up
You can add components to your app using the CLI.
⚠️
Error
Your session has expired. Please sign in again.
✅
Success
Your changes have been saved successfully.
HTML
<div class="alert alert-destructive" role="alert">
  <span class="alert-icon">⚠️</span>
  <div>
    <div class="alert-title">Error</div>
    <div class="alert-desc">Session expired.</div>
  </div>
</div>

/* Tokens: --destructive, --border, --background */
✓ Do
Add role="alert" so screen readers announce the message immediately.
Use for in-context messages related to the current form or view — not for global notifications.
Keep title short (3–5 words) and put detail in the description.
✕ Don't
Don't use Alert for transient notifications — use Toast instead.
Don't stack multiple alerts — consolidate into one or use a list inside the description.
Don't use Destructive for warnings — reserve it for errors only.

Separator

Display

Divides content visually or semantically. Horizontal or vertical.

Section A
Section B
HTML
<!-- Horizontal -->
<hr class="separator-h" />

<!-- Vertical (inside a flex row) -->
<div class="separator-v" aria-orientation="vertical" />

/* Uses: --border token */
.separator-h {
  height: 1px;
  background: hsl(var(--border));
}
✓ Do
Use <hr> for semantic separation of content sections.
Add aria-orientation="vertical" for vertical separators in flex layouts.
✕ Don't
Don't overuse separators — white space (margin/padding) should do most of the separating work.
Don't add separators between every item in a dense list — use subtle row hover instead.

Progress

Display

Communicates task completion. Always pair with a percentage or label.

Uploading…60%
HTML
<div
  class="progress-track"
  role="progressbar"
  aria-valuenow="60"
  aria-valuemin="0"
  aria-valuemax="100"
  aria-label="Upload progress"
>
  <div class="progress-fill" style="width: 60%"></div>
</div>
✓ Do
Add role="progressbar" with aria-valuenow/min/max for screen readers.
Show a percentage label alongside the bar so users don't have to estimate visually.
✕ Don't
Don't use Progress for indeterminate states — use a Skeleton or spinner instead.
Don't jump the progress bar backwards — it confuses users and breaks trust.

Skeleton

Display

Loading placeholder that mirrors the shape of incoming content.

HTML
<!-- Mirrors the shape of a user card -->
<div aria-busy="true" aria-label="Loading content">
  <div class="skeleton"
    style="width:40px;height:40px;border-radius:9999px">
  </div>
  <div class="skeleton"
    style="height:14px;width:60%">
  </div>
</div>
✓ Do
Match the skeleton shape to the actual content layout (avatar circles, text line widths).
Add aria-busy="true" on the container and remove it once content loads.
✕ Don't
Don't use a spinner when a Skeleton is possible — skeletons reduce perceived loading time.
Don't animate the skeleton indefinitely — replace it with content as soon as data arrives.

Tabs

Navigation

Displays layered content panels, one at a time.

Manage your account settings and profile information.
Change your password and security preferences.
Configure how and when you receive notifications.
HTML
<div role="tablist" class="tabs-list">
  <button role="tab" aria-selected="true"
    aria-controls="panel-account"
    class="tab-trigger active">Account</button>
  <button role="tab" aria-selected="false"
    aria-controls="panel-password"
    class="tab-trigger">Password</button>
</div>
<div role="tabpanel" id="panel-account"> … </div>
✓ Do
Use role="tablist", role="tab", role="tabpanel" with aria-selected and aria-controls.
Support keyboard navigation: Arrow keys switch tabs, Enter/Space activates.
Keep tab labels to 1–2 words; use icons only with a visible label.
✕ Don't
Don't use Tabs for a wizard/multi-step flow — use a stepper component instead.
Don't exceed 5–6 tabs; use a Select or sidebar navigation for more options.
Don't hide critical information behind a non-default tab without a visual cue (badge, dot).

Tooltip

Feedback

Supplementary label for icon-only controls. Shown on hover and focus.

Copy to clipboard
Delete this item permanently
HTML
<div class="tooltip-wrap">
  <div class="tooltip-bubble" role="tooltip" id="tt1">
    Copy to clipboard
  </div>
  <button
    aria-describedby="tt1"
    class="btn btn-md btn-outline"
  >Copy</button>
</div>
✓ Do
Use for icon-only buttons where the action isn't immediately obvious.
Add role="tooltip" and link via aria-describedby.
Keep tooltip text under 10 words — supplementary, never essential.
✕ Don't
Don't put interactive content (links, buttons) inside a tooltip.
Don't use tooltips on mobile — hover doesn't exist. Use a help icon that opens a popover.
Don't duplicate the button's visible label in a tooltip — it's redundant.

Dialog

Overlay

Interrupts the user to request a decision or collect input. Use sparingly.

Edit profile
Update your display name and email address.
HTML
<dialog
  class="dialog-demo"
  role="dialog"
  aria-modal="true"
  aria-labelledby="dlg-title"
>
  <div class="dialog-header">
    <h2 class="dialog-title" id="dlg-title">
      Edit profile
    </h2>
  </div>
  <div class="dialog-footer">
    <button class="btn btn-sm btn-outline">Cancel</button>
    <button class="btn btn-sm btn-primary">Save</button>
  </div>
</dialog>
✓ Do
Use the native <dialog> element — it handles focus trap and Esc automatically.
Add aria-modal="true" and aria-labelledby pointing to the title.
Return focus to the trigger element when the dialog closes.
✕ Don't
Don't open a dialog from within a dialog — flatten to a single multi-step flow.
Don't use dialogs for simple notifications — use Toast or Alert instead.
Don't skip the overlay backdrop — without it, users lose sense of modal context.

Dropdown Menu

Overlay

Contextual list of actions triggered by a button. Supports keyboard shortcuts.

Profile ⇧⌘P
Billing ⌘B
Settings ⌘S
Log out ⇧⌘Q
HTML
<ul role="menu" class="dropdown-demo">
  <li role="menuitem" class="dropdown-item">
    Profile
    <kbd class="dropdown-shortcut">⇧⌘P</kbd>
  </li>
  <li role="separator" class="dropdown-sep"></li>
  <li role="menuitem"
    class="dropdown-item destructive">
    Log out
  </li>
</ul>
✓ Do
Use role="menu" and role="menuitem" so screen readers read it as a menu.
Group related actions with a separator. Put destructive actions last, visually separated.
Close on outside click, Esc, and after any item selection.
✕ Don't
Don't use Dropdown for navigation — use a nav component with links instead.
Don't put more than 8–10 items in a dropdown — use a modal with a list instead.
Don't disable menu items without explaining why (tooltip or section label).

Breadcrumb

Navigation

Shows the user's location in a hierarchy. Always make ancestor links clickable.

Home / Components / Breadcrumb
HTML
<nav aria-label="Breadcrumb">
  <ol class="breadcrumb">
    <li><a class="breadcrumb-item" href="/">Home</a></li>
    <li aria-hidden="true"><span class="breadcrumb-sep">/</span></li>
    <li><span
      class="breadcrumb-item current"
      aria-current="page"
    >Button</span></li>
  </ol>
</nav>
✓ Do
Wrap in <nav aria-label="Breadcrumb"> and use <ol> for semantic order.
Mark the current page with aria-current="page" and make it non-clickable.
Add aria-hidden="true" to separator characters so screen readers skip them.
✕ Don't
Don't show breadcrumbs on top-level pages — only relevant from depth 2+.
Don't truncate crumbs aggressively on mobile — show at least Home → … → Current.

Pagination

Navigation

Navigate between pages of a dataset. Previous/Next text buttons flank numbered page buttons. The active page is indicated with a border — no fill — matching the shadcn token system.

Interactive — click a page
HTML + CSS
<nav aria-label="Pagination" class="pagination">

  <!-- Previous -->
  <button class="pag-nav" aria-label="Previous page">‹ Previous</button>

  <!-- Page numbers -->
  <div class="pag-center">
    <button class="page-btn">1</button>
    <button class="page-btn active" aria-current="page">2</button>
    <button class="page-btn">3</button>
    <span  class="page-ellipsis">…</span>
    <button class="page-btn">10</button>
  </div>

  <!-- Next -->
  <button class="pag-nav" aria-label="Next page">Next ›</button>

</nav>

/* Key tokens */
.page-btn.active { border-color: hsl(var(--foreground)); }
.pag-nav         { border: 1px solid hsl(var(--border)); }
ClassElementDescription
.pagination<nav>Flex container, space-between layout
.pag-nav<button>Previous / Next text buttons with border
.pag-center<div>Wraps page number buttons
.page-btn<button>Individual page number — 36×36px
.page-btn.active—Current page — --foreground border, no fill
.page-ellipsis<span>Non-interactive ellipsis gap indicator
✓ Do
Always add aria-current="page" to the active page button for screen readers.
Use disabled on Previous at page 1 and Next at the last page.
Use the ellipsis <span class="page-ellipsis"> to indicate skipped ranges — always non-interactive.
Pair with a "Page X of Y" label on mobile where space is limited.
✕ Don't
Don't show pagination for fewer than 2 pages — conditionally render it.
Don't mix pagination and infinite scroll in the same list view.
Don't hard-code colors — use --foreground, --border, --accent tokens.
Don't show more than ~7 page buttons without an ellipsis — truncate long ranges.
